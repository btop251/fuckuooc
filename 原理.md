# 技术原理

本文档详细说明脚本的核心实现原理，包括浏览器自动化、视频学习、测验答题流程，以及对多模态大模型的依赖方式。

## 0. 功能边界

当前实现的自动化边界如下：

| 类型 | 支持情况 | 技术说明 |
|---|---|---|
| 视频播放 | 支持✅ | 通过 Playwright 控制页面内 `<video>` 元素播放，并记录已看视频 |
| 视频中弹题 | 支持✅ | 通过穷举选项组合自动提交 |
| 视频后章节测验 | 支持✅ | 对题目截图后调用多模态模型返回 JSON 答案并自动勾选 |
| 讨论 | 支持✅ | 实现讨论区发言、互动、富文本编辑等流程 |
| 考试 | 不支持❌ | 没有实现考试场景识别、考试流程控制和提交逻辑 |
| 作业 | 支持✅ | 自动查找作业目标，发现未完成作业按照时间先后依次完成 |

需要特别说明的是：章节测验正确率并不由脚本保证，而是由模型的图片理解和题意判断能力决定。

## 1. 浏览器自动化（Playwright）

### 1.1 为什么选择 Playwright

脚本使用 [Playwright](https://playwright.dev/) 驱动真实的 Chrome 浏览器，而非发送 HTTP 请求。原因：

- UOOC 的视频播放、章节切换、测验提交都依赖前端 JavaScript（AngularJS）渲染，纯请求无法模拟
- 视频的播放进度由前端 `<video>` 元素上报，必须在真实浏览器环境中执行
- 阿里云智能验证依赖浏览器指纹与行为分析，无头请求会被直接拦截

### 1.2 反检测措施

Playwright 启动的浏览器默认会暴露自动化特征，脚本通过以下手段规避检测：

```javascript
// 1. 移除 navigator.webdriver 标记
Object.defineProperty(navigator, 'webdriver', { get: () => undefined });

// 2. 伪造 Chrome 插件列表
Object.defineProperty(navigator, 'plugins', { get: () => [1, 2, 3] });

// 3. 伪造语言偏好
Object.defineProperty(navigator, 'languages', { get: () => ['zh-CN', 'zh', 'en'] });

// 4. 伪造 window.chrome 对象（正常 Chrome 都有）
window.chrome = { runtime: {} };
```

此外还设置了自定义 User-Agent，并通过 `--disable-blink-features=AutomationControlled` 和 `ignoreDefaultArgs: ['--enable-automation']` 移除 Chrome 的自动化标识栏。

### 1.3 模拟人类操作

对于需要通过行为分析的操作（如智能验证和按钮点击），脚本不使用 Playwright 的直接 `.click()`，而是模拟人类鼠标轨迹：

1. 获取元素的 `boundingBox()`（屏幕坐标）
2. 将鼠标以 5 步渐进移动到目标附近（加入 ±5px 随机偏移）
3. 等待 100~300ms 的随机延迟
4. 在目标中心点击

这种方式比直接调用 DOM click 更不容易被行为分析系统检测。

## 2. 视频播放原理

### 2.1 DOM 注入控制

视频播放通过 `page.evaluate()` 直接操作页面内的 `<video>` DOM 元素：

```javascript
const v = document.querySelector('video');
v.muted = true;           // 静音
v.playbackRate = 2.0;     // 2倍速
if (v.paused) v.play();   // 确保播放
```

`page.evaluate()` 的本质是将 JavaScript 注入到浏览器页面的执行上下文中运行。代码在页面的 `window` 作用域内执行，可以访问所有 DOM API，与用户在控制台手动输入代码效果完全相同。

### 2.2 完成判定

脚本每 2.5 秒轮询一次视频状态，满足以下任一条件即判定完成：

- `video.ended === true`
- 播放进度比例 > 98.5%（`currentTime / duration > 0.985`）
- 剩余时间 < 3 秒

### 2.3 去重机制

每个视频的 URL 经过标准化处理（解码 + 去除 query 参数 + 提取 pathname），存入 `data/<username>/<courseId>.txt`。重新运行时自动加载，已看视频直接跳过。

这里有一个边界需要说明：这份“已看记录”只会记住**由本程序实际播放并记录过**的视频链接。如果某些视频是用户此前在 UOOC 页面里手动观看的，而不是通过本程序观看，那么这些视频链接不会出现在程序的记录文件中。重新运行时，脚本无法识别这部分“人工已看但程序未记录”的视频，因此少数情况下可能会重复观看一小部分视频。

### 2.4 视频内弹窗测验

UOOC 视频播放中途会弹出 `#quizLayer` 测验弹窗（通常是单选/多选题），脚本通过**穷举法**破解：

对 N 个选项，遍历 $2^N - 1$ 种非空组合（用位运算 `i >> j & 1` 控制每个选项的勾选状态），每次提交后检查弹窗是否消失。消失即为正确答案。

## 3. 视频后章节测验答题流程

章节末的测验 / 测试题目处理流程如下：

```
┌──────────┐     ┌──────────────────┐     ┌──────────┐
│ 截图题目    │ ──→ │ 多模态大模型答题 │ ──→ │ 点击选项  │
│ (Playwright) │  │ (LLM_MODEL)      │     │ (DOM操作) │
└──────────┘     └──────────────────┘     └──────────┘
```

### 3.1 截图

使用 Playwright 的 `element.screenshot()` 对每道题的 `.queContainer` 容器截图为 PNG。每个课程使用独立文件名 `image_<courseId>.png` 避免并行冲突。

### 3.2 多模态大模型识图答题

**模型**：默认 `doubao-seed-2-0-mini-260215`（火山方舟 / 豆包）。本项目仅在火山引擎平台 + doubao 系列模型下经过测试，其他模型或平台不保证可用性。理论上支持任何 OpenAI 格式的 API 地址，如遇兼容性问题可尝试手动修改 `utils/module.js` 中的请求参数。

当前实现对模型有明确能力要求，至少需要满足：

- 支持图片输入 / 多模态理解
- 兼容 OpenAI Chat Completions 风格接口
- 支持深度思考（`thinking` 参数）


如果模型缺少其中任意一项能力，脚本就可能失败：

- 不支持图片输入：无法读取题目截图内容
- 不兼容 OpenAI 接口：请求体字段可能不被接受


**API 调用方式**：

脚本使用 OpenAI 兼容格式调用火山方舟 API（默认地址 `https://ark.cn-beijing.volces.com/api/v3/chat/completions`，可通过 `config.txt` 中的 `BASE_URL` 字段切换到任意 OpenAI 兼容平台）：

1. 将截图读取为 base64 编码
2. 以 `image_url` 类型传入多模态消息（data URL 格式）
3. 附加文本指令要求模型阅读图中题目并只返回 JSON 答案

**深度思考**：

默认启用深度思考。通过 `thinking` 参数控制思考行为：

- 首次答题使用 `doubao-seed-2-0-mini-260215`，思考强度为 medium（`budget_tokens: 4096`）
- 重做测验时切换至 `doubao-seed-2-0-lite-260215`，思考强度为 high（`budget_tokens: 16384`）

深度思考可以显著提升数学等推理类题目的正确率，但会增加延迟和 token 消耗。

**浏览器缩放**：

在测验答题前，脚本会将浏览器页面缩放至 75%（`document.body.style.zoom = '0.75'`），以确保题目截图的完整性，避免选项被截断。答题完成后恢复至 100%。

**温度设置**：

模型温度设置为 0，确保输出的确定性和一致性。

### 3.3 重试与降级

部分课程采用**闯关模式**，测验必须达到及格分数（通常 60 或 80 分）才能解锁后续章节。为最大化通过率，脚本采用**三级递进策略**：

#### 第一级：默认模型答题

使用 `config.txt` 中配置的默认模型（`doubao-seed-2-0-mini-260215`，medium 思考强度），逐题截图 → 大模型识别 → 点击选项。每道题的截图 + 识别流程最多重试 5 次；如果 5 次都返回空答案，触发降级机制：**随机选择一个选项**，确保试卷不会留空。全部题目作答完毕后提交试卷。

#### 第二级：强模型整卷重做

提交后，脚本检测页面是否出现未通过提示弹窗（layui dialog，包含"请重新提交测验"字样）。如果检测到该弹窗，说明测验未通过，脚本会：

1. 等待弹窗消失
2. 清空所有已选选项
3. 切换至重试模型 `doubao-seed-2-0-lite-260215`，使用 high 思考强度
4. 重新逐题截图、识别、作答并提交

两级 LLM 尝试合计最多 2 次（`MAX_ATTEMPTS = 2`）。

清空选项的方式：

- 复选框（多选题）：通过 Playwright 的 `uncheck()` 方法取消勾选
- 单选框（单选题/判断题）：通过 `page.evaluate()` 注入 JavaScript，强制将 `radio.checked` 设为 `false`

由于页面基于 AngularJS，直接修改 DOM 的 checked 状态不会触发 Angular 的数据模型更新，但只要重新点击选项即可正常提交。

#### 第三级：暴力遍历兜底（Brute-Force）

如果两次 LLM 尝试均未通过，脚本启动暴力遍历策略 `bruteForceWrongQuestions()`，其核心思路是：**只改错题，穷举所有选项组合，逐组合提交直到通过**。

流程如下：

1. **提取得分**：通过 `getQuestionResults()` 从 DOM 中读取每道题的得分状态（`.scores .color-red` 元素），区分出"已得分"和"未得分"的题目
2. **锁定正确题**：已得分的题目保持原有选项不变，只对未得分题目进行爆破
3. **生成候选组合**：对每道未得分题，根据题型生成所有候选答案：
   - **单选题/判断题**（`radio`）：每个选项值单独构成一组，逐个尝试
   - **多选题**（`checkbox`）：生成所有非空子集，按子集大小排序（2 → 3 → … → 1），优先尝试多选组合（因为多选题答案通常 ≥ 2 个）
4. **逐组合提交**：对于每个候选组合：
   - 通过 `clearSelectionsByName()` 仅清空该题的选项（按 input `name` 属性定位，不影响其他题目）
   - 通过 `applyAnswers()` 在页面及所有 iframe 中点击目标选项
   - 调用 `submitAndCheck()` 提交试卷并检测是否通过
   - 如果整卷通过，立即结束
   - 如果该题得分但整卷未通过，保存该题答案，继续爆破下一道错题
5. **跳过无解题**：如果某道题穷尽所有组合仍未得分，跳过并继续

> **工程事实**：
>
> - 脚本能保证"流程尽量继续"，不能保证"答案一定正确"
> - 即使模型具备多模态理解能力，也可能识别错题、漏看选项、理解偏差或输出空答案
> - 暴力遍历理论上可以找到所有客观题的正确答案，但每次组合都要提交试卷，速度较慢且会产生大量提交记录
> - 选项数过多的多选题（如 6+ 个选项）会产生指数级组合（$2^n - 1$），耗时可能显著增长

### 3.4 智能验证

提交试卷时同样可能触发智能验证，脚本复用登录时的验证码处理逻辑（模拟人类点击），最多重试 10 次。

## 4. 章节导航

### 4.1 DOM 查询

课程的章节列表通过 CSS 选择器 `.oneline` 获取所有可见章节项。当前激活章节通过 `.active` 类名定位。

### 4.2 章节类型判断

进入每个章节后，脚本通过以下逻辑判断内容类型：

- **视频章节**：页面中存在 `<video>` 元素
- **测验/测试章节**：侧边栏标题包含“测验”/“测试”关键词，或页面中出现“提交”/“开始测验”等文字

当前实现已经刻意排除了“作业”作为自动答题目标，原因是：

- 作业页面结构和交互方式不稳定
- 作业可能包含主观题、上传题、文本输入题、附件题等，超出当前脚本能力边界
- 将作业误判为测验会导致文档描述和真实行为不一致

### 4.3 自动推进

学习循环：
1. 检查当前章节是否有未看视频 → 有则播放
2. 检查是否有测验任务点 → 有则答题提交
3. 都没有则自动点击下一章节
4. 到达末尾时结束

如果课程流程中出现讨论、考试、作业等未支持内容，脚本不会可靠完成这些部分，需要人工介入。

## 5. 并行架构

### 5.1 共享登录 + 独立标签页

```
Browser
└── Context (共享 cookies/session)
    ├── Page 1 (登录页 → 课程列表)
    ├── Page 2 (评论与作业窗口)
    ├── Page 3 (课程 A) ──→ VideoTracker A
    ├── Page 4 (课程 B) ──→ VideoTracker B
    └── Page 5 (课程 C) ──→ VideoTracker C
```

登录只做一次，之后每个课程在 `context.newPage()` 创建的独立标签页中运行。Playwright 的 `BrowserContext` 保证所有页面共享同一组 cookies，因此无需重复登录。

### 5.2 并发池

使用 `Promise.race()` 实现简单的并发池控制：

- 维护一个 `pending` 集合，最多容纳 `MAX_CONCURRENT`（默认 3）个正在运行的 Promise
- 当池满时，等待任意一个 Promise 完成后再启动新任务
- 最后用 `Promise.all()` 等待所有剩余任务结束

### 5.3 状态隔离

每个并行课程拥有独立的：
- `VideoTracker` 实例（拥有独立的 `watched` Set）
- 截图文件（`image_<courseId>.png`）
- 日志前缀（`[课程N/总数:ID]`）

文件系统层面的观看记录（`<courseId>.txt`）按课程 ID 天然隔离，不存在并发写入冲突。

## 6. Iframe 处理

UOOC 的登录框和部分页面元素嵌套在 `<iframe>` 中。脚本通过 `locateInAnyFrame()` 遍历主页面和所有子 frame，确保能找到目标元素：

```javascript
// 1. 先在主页面查找
page.locator(selector)

// 2. 若未找到，逐个遍历 iframe
for (const frame of page.frames()) {
    frame.locator(selector)
}
```

这保证了无论元素在主文档还是 iframe 中，都能被正确定位和操作。

## 7. 依赖总结

| 依赖 | 用途 | 必要性 |
|---|---|---|
| Playwright | 浏览器自动化、视频播放、DOM 操作 | 核心，不可替代 |

## 8. 风险与限制

从实现角度看，当前方案存在以下天然限制：

- 强依赖页面 DOM 结构。一旦 UOOC 页面类名、层级、按钮文案发生变化，定位逻辑就可能失效。
- 强依赖模型能力。模型是否支持图片输入和深度思考，直接决定章节测验能否完成。仅在火山引擎 + doubao 系列模型下经过测试，其他模型不保证可用性。
- 不保证答题正确率。脚本只负责自动化流程，无法保证模型识别和推理始终正确。
- 不覆盖主观互动类任务。讨论、考试、作业等场景没有纳入当前自动化设计。
- 会产生外部 API 成本。截图识别、多次重试、并发课程都会增加模型调用费用。

## 9. 已知缺陷

### 9.1 视频记忆缺陷

脚本通过本地文件（`data/<username>/<courseId>.txt`）记录已播放的视频链接。如果某些视频是用户此前在 UOOC 网页上手动观看的（而不是通过本程序），这些记录不会出现在本地文件中。重新运行时，脚本可能会重复播放这部分视频。不影响最终学习进度，只是多花一些时间。

### 9.2 视频内测验前端小 Bug

视频内弹出测验在自动作答后，前端 UI 可能出现轻微显示异常（如选项高亮状态不一致）。这是因为脚本通过 DOM 操作勾选选项，没有触发 AngularJS 的数据绑定更新。此问题仅影响前端显示，不影响答题结果提交和后端记录，可忽略。
